What We Solved

NVIDIA's model.transcribe() is not thread-safe. You can't call it from a server without hitting crashes. We found and fixed 4 distinct bugs, then built a production server on top that actually works under sustained load.

BugSymptomFix
freeze/unfreeze race 97/98 crashes = ValueError on concurrent transcribe() Removed redundant calls — @inference_mode() already handles it
CUDA Graph crash on T4 cudaErrorIllegalAddress → segfault (signal 139) cuda_graphs: false config — Turing arch doesn't support conditional nodes
Cross-thread CUDA tensor GC CachingHostAllocatorImpl::free() segfault under sustained load Serialize results to plain Python dicts on GPU thread
Residual generator leak Same segfault, intermittent — generators outlive function, GC'd on wrong thread gc.disable() + gc.collect(0) per batch on GPU thread
Why 43 not 47? Our earlier 47 RPS result was a race condition — Python's automatic GC happened to not trigger during that 3-minute test window. Under longer sustained load, it crashes. The 43 RPS config has verified crash safety: gc.disable() prevents async-thread GC, and gc.collect(0) on the GPU thread cheaply cleans young objects without scanning the model's millions of gen-2 objects.

Best Config to Run Parakeet Batch in Prod

L4 GPU — Production RECOMMENDED

# serving-batch.yaml — L4 production (43 RPS, $0.50/hr spot) batch_model: name: "nvidia/parakeet-tdt-0.6b-v3" compile: true # +159% serial throughput (322ms → 129ms GPU time) amp: true cuda_graphs: true # RNNT decoder CUDA graphs (L4/A100+ only) batcher: max_batch_size: 32 # With torch.compile, 32 > 96 (compiled kernels cycle faster) max_wait_seconds: 0.002 # Near-instant flush — old 0.1s wasted 100ms idle per batch max_queue_depth: 4096 # Deep queue absorbs bursts (256 caused 503 cascade)
Why batch_size 32 not 96? Without torch.compile, batch=96 was 40% faster than 32. But with torch.compile enabled, the compiled CUDA kernels cycle faster with smaller batches — batch=32 now outperforms batch=96. This was validated in the in-cluster benchmark.

T4 GPU — Budget / Dev / Low-traffic

# serving-batch-t4.yaml — T4 budget (6.9 RPS, $0.35/hr spot) batch_model: name: "nvidia/parakeet-tdt-0.6b-v3" compile: true amp: true cuda_graphs: false # MUST be false — Turing crashes with CUDA graph conditional nodes prefetch: true # +7% throughput, -24% latency (bypasses Lhotse DataLoader overhead) batcher: max_batch_size: 32 max_wait_seconds: 0.002 max_queue_depth: 4096

Parameter Decisions — All Benchmark-Validated

ParameterValueWhy (data)
max_wait_seconds 0.002 #1 throughput lever. 0.1s → 0.002s = +111% RPS. GPU was idling 100ms between every batch. Sweep tested 0.005 through 0.2s.
max_batch_size 32 With torch.compile: 32 beats 96. Without compile: 96 was optimal. Sweep tested 4 through 128.
max_queue_depth 4096 At 256, sustained load caused cascading 503s. 4096 absorbs bursts. Memory cost negligible (metadata only).
compile true Serial GPU time: 322ms → 129ms (-60%). Serial RPS: 2.96 → 7.68 (+159%). One-time 60s warmup.
cuda_graphs true (L4) / false (T4) Turing (T4, compute cap 7.5) crashes with NVRTC-compiled conditional nodes. Ada Lovelace+ is fine.
prefetch false (L4) / true (T4) T4 gains +7% throughput / -24% latency. L4 is already GPU-bound, prefetch doesn't help.
Multi-model pool Don't use L4: 4 copies = 24% slower (splits batches across CUDA streams). T4: 3 copies = OOM at 97% VRAM.
GC strategy gc.disable() + gc(0)/batch + gc/50 5 approaches tested. gc.enable() = segfault. gc/request = 5x slow. gc(0) scans only young objects = nearly free.

49.6 → 42.83 RPS drop explained: The 49.6 run used gc.enable() (Python's default). It worked for 60s but was a ticking time bomb — automatic GC can trigger on the async thread at any time and segfault. We traded ~7 RPS for verified crash safety. The gc.collect(0) per-batch approach adds <1ms overhead vs. full gc.collect() which adds ~1.2s.

Bottom Line

For production: Use L4 with the serving-batch.yaml config above. 43 RPS sustained, 0 failures, $0.0032/1K requests, scales horizontally to 172 RPS with 4 pods. Every parameter is backed by benchmark data, not defaults.

The key insight: NeMo's transcription path has 4 thread-safety bugs that make it unusable in a server without these fixes. The hardest to solve was the GC segfault — Python's automatic GC can free CUDA pinned-memory tensors on the wrong thread. We went through 5 GC strategies to find one that's both crash-safe AND fast.
